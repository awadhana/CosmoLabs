# CEO Plan: Autonomous Client-Site Build Pipeline ("Forge")

Generated by /plan-ceo-review · 2026-07-09 · Mode: **EXPANSION**
Repo (marketing): awadhana/CosmoLabs (public) · Pipeline repo: **new private repo**
Revised after adversarial spec review (v2: feasibility fixes applied).

## What this is

Turn the CosmoLabs intake into a closed loop: a client submits a brief, you approve
it in Discord, and a GitHub Actions job runs an autonomous agent that builds a unique,
animated, media-rich website from that brief, tests it, deploys a preview to Vercel,
and (after your second approval) promotes it to production and emails the client.

This is not a new idea bolted on. `ARCHITECTURE.md` already specifies this exact
consumer ("Hermes / Obsidian pipeline ... kick off the build agent"). The producer
(intake → Blob → admin listing) is done. This is **Phase 2**. Forge **supersedes** the
documented `new→processing→done` lifecycle with a richer state set (below).

## Decisions locked (this review)

| # | Decision | Choice | Why |
|---|----------|--------|-----|
| D1 | Generation strategy | **Full agent build** | Max uniqueness/moat; agent writes each site from scratch |
| D2 | Front gate | **Discord approve-to-start** (waits) | Cost + brand control before any compute is spent |
| D3 | Build host | **GitHub Actions** | Matches the "github workflow" vision |
| D4 | Pipeline repo | **Private** | Client PII must never hit world-readable Actions logs |
| — | Review mode | **EXPANSION** | Build the ambitious version; flag oceans |

## Defaults chosen for you (with the alternative)

| Area | Default | Alternative |
|---|---|---|
| Generation engine | Claude Code headless (`claude-code-base-action` / `claude -p`) running the **ui-ux-pro-max** skill vendored into the runner | Single large structured LLM call |
| Model | `claude-opus-4-8` for design/build, `claude-haiku-4-5` for cheap sub-steps | Opus everywhere |

| Site hosting | **Previews stay as Vercel preview URLs**; **go-live provisions a per-client Vercel project/alias** (go-live is per-client, by design) | Per-client project from the first build |

| Notify trigger | **`intake.js` calls `/api/pipeline/notify` inline** after the Blob write, best-effort like email (no HTTP hop) | Vercel Cron poll of the admin endpoint |

## Architecture

```
┌────────────── PUBLIC (marketing repo, Vercel) ──────────────┐
│  index.html intake ─▶ POST /api/intake ─▶ Blob submission.json (status:"new") │
│                                    │  (intake.js fires this inline, best-effort)│
│                                    ▼                          │
│                     NEW: POST /api/pipeline/notify           │
│                       └▶ Discord webhook message:            │
│                          brief summary (title/desc only) +   │
│                          HMAC-signed Approve URL + Skip URL   │
└──────────────────────────────────────┬──────────────────────┘
                                        │ you click the signed Approve link
                                        ▼
┌────────── NEW: GET /api/pipeline/approve?sig=... (HMAC + TTL) ┐
│  verify sig+expiry; check status=="new" (idempotent);         │
│  set status:"approved" ─▶ GitHub API repository_dispatch      │
│    { event_type:"build_site", client_payload:{ id } }         │
│  (needs GITHUB_DISPATCH_TOKEN as a Vercel env var)            │
└──────────────────────────────────────┬──────────────────────┘
                                        ▼
┌────────── PRIVATE pipeline repo · GitHub Actions ───────────┐
│  build-client-site.yml (repo secrets: BLOB_READ_WRITE_TOKEN, │
│    ANTHROPIC_API_KEY, PEXELS_KEY, VERCEL_TOKEN, HMAC_SECRET,  │
│    DISCORD_WEBHOOK)                                           │
│   1 fetch brief by id; STRIP PII (drop email/phone/ip/UA);   │
│     hand agent only title/desc/refs/colors/clarifications/files│
│   2 Claude Code headless + ui-ux-pro-max → design system     │
│     (transcript suppressed / redirected to scrubbed artifact) │
│   3 agent writes site from scratch into a scratch dir        │
│   4 pull + optimize stock media (Pexels) → self-host         │
│   5 TEST GATE (Playwright) → auto-fix loop (3 attempts total) │
│   6 vercel deploy (preview); write vercelDeploymentId +       │
│     projectId + previewUrl back into submission.json          │
│   7 status:"preview"; Discord: preview URL + signed Approve/  │
│     Request-changes links                                     │
└──────────────────────────────────────┬──────────────────────┘
                                        │ you click signed Approve (gate 2)
                                        ▼
   POST /api/pipeline/promote: provision per-client Vercel project/alias,
   promote the deployment, status:"done", email client (new sendSiteLiveEmail)
```

### State machine (Forge state set — supersedes `new→processing→done`)

```
 new ──approve(link)──▶ approved ──dispatch──▶ building
   │                       │                      │
 skip                  (dispatch fail)      ┌──────┴────────┐
   ▼                       ▼            all tests pass   fail after 3 attempts
 dropped                approved          ▼                  ▼
                        (re-runnable)   preview          build_failed ──▶ Discord alert
                                          │ ▲                (re-runnable)
                             approve──────┘ │request-changes──▶ building
                                ▼
                              done ──▶ email client
```

Owner-per-transition (avoids the last-writer-wins race on `submission.json`):
marketing endpoints own `new/approved/done`; the Actions runner owns
`building/preview/build_failed`. Each writer checks current status before writing
(idempotent). The linear state machine bounds the race.

## Agent step in CI (the riskiest assumption, now specified)

- **Provisioning:** ui-ux-pro-max is on the laptop, not the runner. Vendor it into the
  pipeline repo under `.claude/skills/ui-ux-pro-max/` (committed, or a git submodule /
  npm package). If the skill uses the shadcn MCP server, configure that MCP in the
  runner too.
- **Invocation:** `claude-code-base-action` (or `claude -p`) with `ANTHROPIC_API_KEY`,
  `--allowed-tools` limited to file writes in the scratch dir + the skill, an explicit
  prompt that passes the PII-stripped brief and says "use the ui-ux-pro-max skill."
- **IO contract:** agent writes the site into `./build/`; the workflow collects that dir.
- **Fix loop:** on a failed test, a second agent turn gets the current `./build/` + the
  specific Playwright/Lighthouse failures and edits in place. Cap: **3 attempts total**
  (1 initial + 2 fixes).
- **Cost ceiling:** there is no clean hard token kill in Claude Code — enforce cost via
  **`--max-turns` + a 20-min wall-clock**. Note the wall-clock is billable Actions
  minutes, so validate real cost/time on one build before trusting estimates.

## Error & Rescue Map

| Codepath | What goes wrong | Rescue | You/client sees |
|---|---|---|---|
| `/api/pipeline/notify` | Discord 5xx/timeout | retry x2 backoff; still store status, log | Approve later via admin fallback list |
| `/api/pipeline/approve` | Forged/replayed link | **HMAC sig + expiry check**; status must be `new` (idempotent) → else 409 | — |
| approve → dispatch | GitHub API 4xx/5xx / billing lock / bad token | retry x2; on fail revert to `approved`, Discord alert | "couldn't start build" ping |
| Actions: fetch brief | Blob 404 / token invalid | fail fast, `build_failed`, alert | "brief not found" |
| Actions: agent build | timeout (>20 min) or max-turns hit | wall-clock kill; `build_failed`; keep scrubbed logs | "build timed out" |
| Actions: agent build | broken/invalid site | **test gate catches it**; fix loop (3 attempts) then `build_failed` | test gate is the backstop |
| Actions: media pull | Pexels 429 / no results | fall back to a curated built-in media set | site still builds |
| Actions: test gate | Playwright flake | retry the failing check once | — |
| Actions: deploy | Vercel 429/5xx | retry x2 backoff; on fail `build_failed` | "deploy failed" alert |
| promote → prod | Vercel provision/alias fails | retry; keep preview live; alert | preview URL still valid |
| email client | SMTP down | best-effort (never throws) | logged; notify manually |

**No catch-all.** Every failure is named, alerts Discord, and leaves the job re-runnable.

## Threat Model

| Threat | Likelihood | Impact | Mitigation |
|---|---|---|---|
| **Client PII leaks in Actions logs** | High if unhandled | High | Private repo (D4) **+ strip PII before the agent/model call** (drop email/phone/ip/userAgent; pass only title/desc/refs/colors/clarifications/files) **+ suppress the agent transcript** or write it to a scrubbed, access-controlled, expiring artifact. (Correction: the brief is data, not a GitHub secret — it is NOT auto-masked.) |
| Forged approve → cost drain | Med | Med | HMAC-signed URL, short TTL, single-use via the status check |
| Prompt injection via the brief | Med | Med | Treat the brief as untrusted data, not instructions; agent tools scoped to the scratch dir; no shell exec from brief content |
| Malicious uploaded file | Low | Med | Already magic-byte-checked at intake; never execute; re-validate in runner |
| Secrets exposure | Low | High | GitHub encrypted secrets (these *are* masked); least-privilege tokens (`GITHUB_DISPATCH_TOKEN` scoped to `actions:write` on the one repo); rotate |
| Generated site ships unsafe content | Low | Med | Trimmed CSP + headers on generated sites; test gate checks obvious injection |
| Runaway cost | Med | Med | `--max-turns` + 20-min wall clock + max 3 attempts + concurrency ≤ 2; Discord gate stops junk before spend |
| Client's unreleased preview crawlable | Med | Low | `X-Robots-Tag: noindex` + unguessable preview URL; optional basic-auth |

## Test Gate spec — "passing" is defined

A build reaches the preview gate only if all HARD checks pass (Playwright, headless):
- [ ] `vercel build` succeeds (valid HTML/CSS/JS)
- [ ] Homepage 200, no uncaught JS console errors
- [ ] All internal links/anchors resolve
- [ ] All media have `src`, load, and carry `alt`
- [ ] No horizontal scroll at 375 / 768 / 1440 px
- [ ] **Lighthouse a11y ≥ 90** (hard)
- [ ] No placeholder leakage (`lorem ipsum`, `TODO`, `{{`, empty sections)
- [ ] CTA/contact present and points somewhere valid

**Advisory (logged, does NOT block):** Lighthouse performance. Rationale: D1 wants
animated, video-rich sites, which routinely miss mobile perf≥80 — blocking on it would
send most builds into the fix loop and fail them. Track it as a trend, tune later.

Fail → fix loop (3 attempts) → `build_failed` + Discord alert.

## Cost & Observability

- **Cost ceiling:** `--max-turns` + 20-min wall clock + max 3 attempts. **Validate real
  cost/time on ONE build before committing an estimate** — agentic Opus from-scratch
  builds burn tokens fast, and wall-clock counts as billable Actions minutes even while
  waiting on the model.
- **Actions-minute reality:** ~2,000 free private-repo min/mo ÷ ~25 min/build ≈ **~80
  builds/month free**, not effectively unlimited. Plan volume against that.
- **Discord gate = the cost valve:** nothing spends until you approve.
- **Observability:** structured PII-scrubbed log line per stage (id, stage, duration,
  outcome); Discord gets build-started / test pass-fail / preview-ready / deploy-done /
  any failure. Weekly summary (builds, pass rate, avg cost) is a fast follow.

## Deployment & Rollout — phased

- **Phase 0 (~1 day):** wire the whole loop with a fixed template the agent only
  re-themes. Proves Discord → approve → Actions → test → Vercel → notify at low variance.
- **Phase 1 (~2-3 days):** swap in the from-scratch agent build behind the same gates +
  test gate + fix loop. Validate real cost/time here.
- **Phase 2 (~1-2 days):** Pexels media (self-hosted), per-client Vercel project on
  go-live, preview `noindex`, cost/observability.
- **Phase 3 (north star):** client self-serve "iterate in chat," one-click go-live.

Safe by construction: two human gates + the test gate mean a bad build never reaches a
client. Rollback = don't promote the preview.

## Engineering review decisions (v3)

- **Sequencing (D1): spike the agent-in-CI assumption FIRST** (half a day) before building
  any pipeline. Run `claude -p` + vendored ui-ux-pro-max locally on one real brief; measure
  tokens, wall-clock, and output quality; confirm headless works and whether the shadcn MCP
  server is needed. Only then build Phase 0. This is the new **T-0 (P1)**, before T0.
- **Secret blast radius (D2): split the Actions workflow into two jobs.** Job 1 `fetch` holds
  `BLOB_READ_WRITE_TOKEN`, strips PII, outputs only the sanitized brief + files. Job 2 `agent`
  runs with NO Blob/Vercel token, tools scoped to file writes in a scratch dir, no shell /
  network egress. A hijacked agent cannot reach client data or deploy. [Layer 1: least privilege.]
- **Write race (D3): add `concurrency: { group: build-${id}, cancel-in-progress: true }`** to
  the build workflow so duplicate dispatches for one submission collapse to one run. Accept the
  bounded read-then-write Blob window (low volume, human-paced, Discord gate catches junk).
  [Layer 1: Actions built-in.]

## Spike results (T-0 executed 2026-07-10)

Ran the agent-build assumption end to end locally: sample brief ("Meridian Coffee Roasters")
→ ui-ux-pro-max design query → generated a full self-contained landing page
(`spike/meridian-coffee.html`, 402 lines) → served on localhost → visual QA in Chrome.

**Outcome: the assumption holds.** The agent produced a genuinely premium, on-brand,
animated, accessible page (editorial Fraunces type, espresso/copper/cream palette, a
CSS coffee-cup with animated steam, an interactive cadence→price toggle, single-origin
cards with tasting notes, a validating signup). Zero console errors.

Three findings that feed the build:
1. **The generic ui-ux-pro-max `--design-system` output was OFF-BRAND** — it returned a
   black + hot-pink "Liquid Glass" system and self-flagged poor a11y/perf. The agent must
   treat the CLIENT brand as authoritative and use the tool for structure/type only, not
   palette. Action: bake this instruction into the T4 build prompt.
2. **A real CSS bug (collapsed `aspect-ratio` hero visual) was INVISIBLE without a live
   render** — it passed structural checks but the cup did not paint; only opening it in a
   browser caught it. This validates the Playwright/visual test gate AND the preview-approve
   gate as essential, not optional. (Bug fixed: definite `width` so `aspect-ratio` yields height.)
3. **Cost/time revised up:** this was one clean pass with no fix-loop. A real CI run
   (design → build → test → 1-2 fixes on Opus) is realistically ~8-20 min and ~$1-4/site.
   Confirm on the first real headless run; the ~80-builds/month free-Actions ceiling is real.

Net: generation quality gets you ~85%; the two gates cover the last 15%. Build with
confidence, keep both gates.
